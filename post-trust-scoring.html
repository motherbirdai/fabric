How our trust scoring works — six signals, no black boxes

Every AI service indexed by Fabric receives a score from 0.00 to 5.00. The score is computed from six independently weighted signals, all derived from public data. No manual reviews. No paid placements. No provider involvement required. Here's exactly how it works.

The problem with single metrics

The MCP ecosystem is growing fast. Thousands of AI services, tools, and models are published every week across npm, PyPI, HuggingFace, and MCP registries like ClawHub. The question every agent developer faces: which ones can you trust?

Download count is the most common proxy for trust. It's also the easiest to game. The ClawHub malware incident proved it — the #1 most-downloaded skill on the platform was actively stealing SSH keys, crypto wallets, and browser passwords. It had thousands of installs. It looked legitimate. And it was malware.

GitHub stars, npm downloads, HuggingFace likes — individually, any of these can be inflated. You need multiple independent signals that are hard to fake simultaneously. That's the core design principle behind Fabric's scoring.

Six signals, weighted by risk

Each signal scores 0–5 independently, then is multiplied by its weight. The six weighted contributions sum to the final score. Security and operational signals carry 65% of the total because they directly affect safety. Adoption and reputation are supporting evidence, not primary drivers.

1. Vulnerability & Safety ×0.25
max 1.25
The highest-weighted signal. Scans for known CVEs (severity-weighted), flagged dependencies, install scripts, typosquatting indicators, and malware signatures across the full dependency tree. Starts at 5.0, deducts per finding. Critical unpatched CVE = floor at 1.0. Active malware = zero.
2. Operational Health ×0.20
max 1.00
The only signal that requires Fabric's own infrastructure. We actively monitor services for uptime, latency, and behavioural consistency. 99.9%+ uptime and sub-500ms p50 = 5.0. Heaviest penalty is for behavioural inconsistency — different responses to identical inputs.
3. Maintenance Activity ×0.20
max 1.00
Is anyone home? Evaluates recency of last commit, 90-day commit frequency, issue response time, PR merge velocity, and release cadence. Active project with regular releases = 5.0. No commits in 12+ months = 1.0 or below.
4. Adoption ×0.15
max 0.75
Logarithmic scale against ecosystem peers. Combines download counts, stars, forks across registries. Normalised by category so niche tools aren't penalised. Download velocity matters more than absolute count. This is the one signal attackers can inflate — that's why it's weighted low.
5. Transparency ×0.10
max 0.50
Checklist-based: open source, recognised licence, README quality, SECURITY.md, published schemas, model cards, API docs, research links. Each element adds to the score. Closed-source with no documentation scores near zero.
6. Publisher Trust ×0.10
max 0.50
Account age, organisational backing, other maintained packages, cross-registry identity consistency, verified domain. New anonymous accounts start near 1.0. Established orgs with clean track records score above 4.0.

The formula

trust_score =
  (vulnerability_safety × 0.25)
+ (operational_health   × 0.20)
+ (maintenance_activity × 0.20)
+ (adoption             × 0.15)
+ (transparency         × 0.10)
+ (publisher_trust      × 0.10)

Worked example

// A well-maintained open source image generation API
trust_score = (4.8 × 0.25) + (4.5 × 0.20) + (3.4 × 0.20) + (2.1 × 0.15) + (3.7 × 0.10) + (4.3 × 0.10)
            = 1.20 + 0.90 + 0.68 + 0.32 + 0.37 + 0.43
            = 3.90 // ✅ Trusted

Three tiers

Every score maps to one of three tiers that determine default agent behaviour:

3.34 – 5.00 Trusted Low risk across all signals. Agents can auto-approve without human confirmation.
1.67 – 3.33 Caution Some signals flagged. Agents should request human confirmation before proceeding.
0.00 – 1.66 Blocked Significant risks detected. Agents deny by default. Manual override required.

Score modifiers

Two additional multipliers can reduce a score below what the signals alone would produce. Providers with fewer than 10 recorded transactions receive a "new provider" penalty of 0.8× on their final score. Providers inactive for 7+ days are flagged with an "inactive" multiplier of 0.7×. Both exist to prevent trust-before-evidence — you shouldn't get a high score just because nothing bad has happened yet.

Data freshness

The scoring engine runs continuously against 10,000+ indexed services. Full dataset refresh runs daily via batch. Event-driven re-checks fire on three conditions: a new version release, a new CVE disclosure affecting the service's ecosystem, or a user search that returns stale data.

Vulnerability data refreshes most aggressively — hourly syncs against advisory databases, with immediate re-scoring when a relevant CVE is published. Transparency and publisher trust change slowly and only need daily checks. Operational Health runs on a 15-minute cycle for high-traffic services.

Custom weights

Pro and Team plans can override default weights. If your use case prioritises speed over security, or cost over quality, you can rebalance the formula without changing the underlying data.

{
  "trustWeights": {
    "vulnerabilitySafety": 0.40,   // security-first config
    "operationalHealth":  0.25,
    "maintenance":        0.15,
    "adoption":           0.10,
    "transparency":       0.05,
    "publisherTrust":     0.05
  }
}

Why this works

Spoofing one signal is easy. Download counts can be inflated overnight. GitHub stars can be purchased. A README can be polished in an hour. But spoofing all six simultaneously — passing vulnerability scans, maintaining consistent uptime, building genuine commit history, publishing from a verified org, showing transparent documentation, and exhibiting normal network behaviour — is almost impossible.

That's the design principle. Not any single signal. The combination.

No manual reviews. No paid placements.

Trust scores are fully automated from public sources. Providers cannot pay for a higher score, request manual review, or influence their ranking. The only way to improve a score is to improve the underlying signals.
