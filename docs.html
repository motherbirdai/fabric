Trust Score

The Fabric Trust Score is the core of the Trust Index. It determines whether a service is trusted, flagged for caution, or blocked outright.

Every AI service indexed by Fabric receives a composite score from 0.00 to 5.00, computed from six independently weighted signals. Each signal scores 0–5 individually, then is multiplied by its weight. The six weighted contributions sum to the final score.

All scoring data is generated automatically from public sources. No manual reviews. No paid placements. No provider involvement required. A provider doesn't need to opt in — or even know they're being scored — for Fabric to evaluate them.

Why six signals? Any single metric can be gamed. Download counts can be inflated. GitHub stars can be bought. Even vulnerability databases have blind spots. By combining six independent signals across security, operations, maintenance, adoption, transparency, and publisher identity — an attacker would need to simultaneously spoof all of them to pass. That's the point.

The six signals

1. Vulnerability & Safety ×0.25

The highest-weighted signal. Measures known security vulnerabilities, malware indicators, and dependency chain risk across the package's entire tree.

Fabric scans for open CVEs (weighted by severity), flagged dependencies, presence of install scripts that execute on download, typosquatting indicators, and known malware signatures. The scan covers the full dependency tree — not just the top-level package.

Scoring starts at 5.0 and deducts based on findings. Zero vulnerabilities and a clean dependency chain scores a full 5.0. A critical unpatched CVE floors the score at 1.0 or below. Active malware detection results in a zero.

Supply chain depth matters. A package with zero direct vulnerabilities can still score poorly if it depends on a library that depends on a library with a critical CVE. Fabric traces the full tree.

2. Operational Health ×0.20

The only signal that requires Fabric's own infrastructure. We actively monitor services for uptime, latency, and behavioral consistency — no provider can fake this one.

Scoring is based on rolling 30-day metrics. 99.9%+ uptime and sub-500ms p50 latency scores 5.0. Each downtime incident deducts based on duration and frequency. Latency spikes above p99 thresholds reduce the score. The heaviest penalty is for behavioral inconsistency — when a service returns different responses to identical inputs across checks.

Services with no endpoint to monitor (CLI tools, local packages) receive minimal signal here, which naturally suppresses their overall score unless other signals compensate.

3. Maintenance Activity ×0.20

Is anyone home? Abandoned software is a security risk regardless of how clean the code is today. This signal captures the human factor behind a project.

Fabric evaluates recency of last commit, commit frequency over a rolling 90-day window, median time to first response on issues, ratio of open to closed issues, pull request merge velocity, and release cadence consistency. An actively maintained project with regular releases scores 5.0. No commits in 12+ months drops to 1.0 or below.

Fabric accounts for false positives — some stable, mature projects have low commit frequency by design. Release cadence and issue responsiveness carry more weight than raw commit volume.

4. Adoption ×0.15

Not a security signal per se, but a strong proxy for how many eyes are on the project. High adoption means more users finding and reporting issues, more pressure to maintain quality, and a larger blast radius if something goes wrong — which motivates faster fixes.

Fabric uses a logarithmic scale against ecosystem peers. A package with millions of weekly downloads scores differently than one with fifty. But raw numbers aren't the whole story — Fabric normalises against category averages so niche tools aren't penalised simply for being specialised. Download velocity (growth trend) matters more than absolute count.

Adoption is gameable. This is the one signal attackers can inflate. The ClawHub malware incident proved it — "What Would Elon Do" hit #1 by downloads despite being active malware. That's why adoption is weighted at only 0.15 and the other five signals independently flagged it.

5. Transparency ×0.10

Can you see what it does before you run it? Open source projects with clear documentation, proper licensing, and published schemas are inherently more trustworthy than black boxes.

Fabric uses a checklist-based approach: is the source code public? Is there a recognised licence? Does the README exceed a meaningful length with code examples? Is there a SECURITY.md? Are input/output schemas published? Is there a model card, API documentation, or linked research? Each element contributes to the score, capping at 5.0.

This signal is largely static — it only needs re-evaluation when a new version or repository update is detected.

6. Publisher Trust ×0.10

Who built this? A first-time anonymous account publishing a critical infrastructure tool is a very different risk profile than a verified organisation with fifty other maintained packages.

Fabric evaluates account age, organisational membership, number of other maintained packages, consistency of identity across registries, whether a verified domain or email is linked, and any prior security incidents. New anonymous accounts start near 1.0. Established publishers with clean track records across multiple registries score well above 4.0.

Cross-referencing identities across package registries helps detect consistent publishers versus throwaway accounts created specifically to distribute a single package.

Scoring formula

The final score is a weighted sum. Each signal contributes proportionally to its weight — security and operational signals carry 65% of the total because they directly affect safety. Adoption and reputation are supporting evidence, not primary drivers.

formula
trust_score =
  (vulnerability_safety × 0.25)
+ (operational_health   × 0.20)
+ (maintenance_activity × 0.20)
+ (adoption             × 0.15)
+ (transparency         × 0.10)
+ (publisher_trust      × 0.10)

// Each signal: 0.00 – 5.00
// Weights sum to: 1.00
// Max possible score: 5.00

Example calculation

example
// A well-maintained open source image generation API
trust_score = (4.8 × 0.25) + (4.5 × 0.20) + (3.4 × 0.20) + (2.1 × 0.15) + (3.7 × 0.10) + (4.3 × 0.10)
            = 1.20 + 0.90 + 0.68 + 0.32 + 0.37 + 0.43
            = 3.90 // ✅ Trusted

Custom weights

Different use cases may call for different priorities. If your workflow values security over speed, or cost over quality — the weights can be rebalanced without changing the underlying data.

config
{
  "trustWeights": {
    "vulnerabilitySafety": 0.40,   // boost security
    "operationalHealth":  0.25,
    "maintenance":        0.15,
    "adoption":           0.10,
    "transparency":       0.05,
    "publisherTrust":     0.05
  }
}

Score thresholds

Every indexed service falls into one of three tiers based on its composite score. These labels are shown publicly on the Trust Index.

ScoreStatusMeaning
3.34 – 5.00TrustedLow risk across all signals. Safe to use without additional verification.
1.67 – 3.33CautionSome signals flagged. Human review recommended before granting access.
0.00 – 1.66BlockedSignificant risks detected. Not recommended for use. Manual override required.

Data freshness

The scoring engine runs continuously against 10,000+ indexed AI services. Full dataset refresh runs daily via batch. Event-driven re-checks are triggered by three conditions: a new version release, a new CVE disclosure affecting the service's ecosystem, or a user search that returns stale data.

This keeps scores current without polling every service every five minutes. Most signals (transparency, publisher trust, adoption) change slowly and only need daily checks. Vulnerability data is the most time-sensitive and is refreshed more aggressively — hourly syncs against advisory databases, with immediate re-scoring when a relevant CVE is published.

Operational Health is the exception — Fabric Monitor runs its own pings on a 15-minute cycle for top-traffic services, with daily checks for the long tail. This is Fabric's proprietary data. Nobody else is monitoring AI service behaviour at this scale.

No manual reviews. No paid placements. Trust scores are fully automated. Providers cannot pay for a higher score, request manual review, or influence their ranking. The only way to improve a score is to improve the underlying signals — fix vulnerabilities, maintain the project, improve uptime, verify your identity.
